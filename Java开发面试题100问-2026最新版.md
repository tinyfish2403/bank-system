## 一、Java 基础（15 题）

### 1. HashMap 的底层实现原理是什么？

**答案：**

JDK 8 中 HashMap 采用 **数组 + 链表 + 红黑树** 的结构：

- **哈希计算**：对 key 的 hashCode 进行扰动函数处理（高 16 位与低 16 位异或），再通过 `(n - 1) & hash` 确定数组下标。
- **冲突处理**：相同位置的元素以链表形式存储。当链表长度 **>= 8** 且数组长度 **>= 64** 时，链表转为红黑树（O(n) -> O(log n)）；当红黑树节点数 **<= 6** 时退化为链表。
- **扩容**：默认容量 16，负载因子 0.75。当元素数量超过 `容量 * 负载因子` 时扩容为原来的 2 倍，扩容时通过 `hash & oldCap` 判断元素是在原位置还是 `原位置 + oldCap`。
- **线程安全**：HashMap 非线程安全，多线程环境下应使用 ConcurrentHashMap。

### 2. ConcurrentHashMap 在 JDK 8 中是如何保证线程安全的？

**答案：**

JDK 8 的 ConcurrentHashMap **放弃了分段锁（Segment）**，改用 **CAS + synchronized** 实现：

- **数据结构**：与 HashMap 相同，数组 + 链表 + 红黑树。
- **put 操作**：
  1. 若当前位置为空，通过 CAS 操作写入新节点。
  2. 若当前位置有节点，使用 `synchronized` 锁住链表头节点（或红黑树根节点），遍历链表/树进行插入或更新。
  3. 扩容时多线程协助迁移（通过 `transferIndex` 分配迁移区间）。
- **size 计算**：使用 `baseCount` + `CounterCell[]`（类似 LongAdder），减少竞争。
- **优势**：锁粒度从 Segment（默认 16 个段）细化到单个桶，并发度大幅提升。

### 3. `==` 和 `equals()` 的区别是什么？`hashCode` 和 `equals` 的关系？

**答案：**

- **`==`**：比较基本类型时比较值，比较引用类型时比较内存地址（是否为同一对象）。
- **`equals()`**：Object 类默认实现等同于 `==`，但 String、Integer 等类重写了 equals，用于比较内容是否相等。
- **hashCode 与 equals 的约定**：
  - 两个对象 `equals()` 为 true，则 `hashCode` 必须相同。
  - `hashCode` 相同，`equals()` 不一定为 true（哈希冲突）。
  - 重写 `equals()` 必须同时重写 `hashCode()`，否则在 HashMap/HashSet 中无法正确工作。

### 4. String、StringBuilder、StringBuffer 的区别？

**答案：**

| 特性 | String | StringBuilder | StringBuffer |
|------|--------|---------------|--------------|
| **可变性** | 不可变（final char[] / JDK 9+ byte[]） | 可变 | 可变 |
| **线程安全** | 安全（不可变） | 不安全 | 安全（synchronized 修饰方法） |
| **性能** | 拼接产生新对象，性能差 | 最快 | 略慢（同步开销） |
| **使用场景** | 少量操作 | 单线程字符串拼接 | 多线程字符串拼接 |

**补充**：JDK 9 将 String 底层存储从 `char[]` 改为 `byte[]` + `coder`（编码标志位），节省拉丁字符的内存占用。

### 5. Java 中的多态是什么？实现条件有哪些？

**答案：**

多态指同一方法在不同对象上有不同的行为。实现条件：

1. **继承或实现接口**：子类继承父类或实现接口。
2. **方法重写（Override）**：子类重写父类方法。
3. **父类引用指向子类对象**：`Parent obj = new Child();`

**多态的类型**：
- **编译时多态**：方法重载（Overload），同名方法参数不同。
- **运行时多态**：方法重写 + 动态绑定，JVM 在运行时确定调用哪个方法（invokevirtual 指令）。

### 6. Java 中的异常体系是怎样的？Checked 和 Unchecked 异常有什么区别？

**答案：**

```
Throwable
├── Error（不可恢复：OutOfMemoryError、StackOverflowError）
└── Exception
    ├── Checked Exception（编译时异常：IOException、SQLException）
    └── RuntimeException（运行时异常：NullPointerException、ClassCastException）
```

- **Checked Exception**：编译时强制要求处理（try-catch 或 throws），通常是可预期的外部异常。
- **Unchecked Exception**：运行时异常，编译器不强制处理，通常是编程错误。
- **最佳实践**：业务异常用自定义 RuntimeException，避免过度使用 Checked Exception 污染方法签名。

### 7. Java 8 的 Stream API 有哪些常用操作？并行流需要注意什么？

**答案：**

- **中间操作**（返回 Stream）：`filter`、`map`、`flatMap`、`sorted`、`distinct`、`limit`、`skip`。
- **终结操作**：`forEach`、`collect`、`reduce`、`count`、`min/max`、`anyMatch/allMatch`、`findFirst`。
- **并行流**（`parallelStream()`）：
  - 底层使用 ForkJoinPool，适合数据量大、计算密集型的场景。
  - **注意**：线程安全问题（避免在 lambda 中修改共享变量）、数据量小时并行开销反而更大、共享线程池可能互相阻塞。
  - 使用 `Collectors` 的线程安全收集器（如 `toConcurrentMap`）。

### 8. Java 的反射机制是什么？有什么应用场景？性能如何？

**答案：**

反射允许程序在运行时获取类的信息（字段、方法、构造器），并动态创建对象、调用方法。

**核心类**：`Class`、`Field`、`Method`、`Constructor`。

**应用场景**：
- Spring IoC 容器（通过反射创建 Bean）
- MyBatis 动态代理 Mapper 接口
- JSON 序列化/反序列化
- 注解处理器

**性能问题**：
- 反射涉及动态解析、安全检查，比直接调用慢数倍。
- **优化**：使用 `setAccessible(true)` 关闭安全检查；缓存反射对象；高频场景考虑使用 MethodHandle 或代码生成（CGLIB、ASM）。
- JDK 17+ 的强封装机制限制了反射访问非公开成员。

### 9. Java 的泛型是什么？类型擦除会带来什么问题？

**答案：**

泛型提供编译时类型安全检查，避免强制类型转换。

**类型擦除**：Java 泛型在编译后擦除类型信息，`List<String>` 和 `List<Integer>` 在运行时都是 `List`。

**带来的问题**：
1. 不能使用基本类型作为类型参数（`List<int>` 不合法）。
2. 不能创建泛型类型的实例（`new T()` 不合法）。
3. 不能创建泛型数组（`new T[]` 不合法）。
4. 不能对泛型类型使用 `instanceof`。

**通配符**：`<? extends T>`（上界，只读）、`<? super T>`（下界，只写）。遵循 PECS 原则（Producer Extends, Consumer Super）。

### 12. Java 的 SPI 机制是什么？和 API 有什么区别？

**答案：**

**SPI（Service Provider Interface）** 是一种服务发现机制：

- **API**：实现方提供接口和实现，调用方直接使用。
- **SPI**：调用方定义接口规范，实现方提供具体实现。通过 `META-INF/services/` 目录下的配置文件发现实现类。

**应用场景**：
- JDBC 驱动加载（`java.sql.Driver`）
- SLF4J 日志框架绑定
- Dubbo 的扩展点加载
- Spring Boot 的自动装配（`spring.factories` / `AutoConfiguration.imports`）

### 13. Java 中如何实现不可变类（Immutable Class）？有什么好处？

**答案：**

**实现步骤**：
1. 类声明为 `final`，防止被继承。
2. 所有字段声明为 `private final`。
3. 不提供 setter 方法。
4. 构造器中深拷贝可变对象引用。
5. getter 方法返回可变字段的防御性拷贝。

**好处**：
- 天然线程安全，无需同步。
- 可作为 HashMap 的 key。
- 简化并发编程，适合函数式编程风格。

**示例**：`String`、`Integer`、`LocalDate`、`Record`（JDK 16+）。

### 14. Java 中的代理模式有哪些？静态代理和动态代理的区别？

**答案：**

| 特性 | 静态代理 | JDK 动态代理 | CGLIB 代理 |
|------|---------|-------------|-----------|
| **实现方式** | 手动编写代理类 | `Proxy.newProxyInstance()` | 生成目标类的子类 |

### 15. Java 中的序列化方式有哪些？serialVersionUID 的作用是什么？

**答案：**

**常见序列化方式**：
1. **Java 原生序列化**（`Serializable`）：效率低、安全性差、不跨语言，不推荐。
2. **JSON**（Jackson、Gson、Fastjson）：可读性好、跨语言，体积较大。
3. **Protobuf**（Google）：二进制、体积小、效率高、跨语言，需定义 `.proto` 文件。
4. **Hessian**：二进制、跨语言，Dubbo 默认序列化。
5. **Kryo**：高性能 Java 序列化，Spark/Flink 中使用。

**serialVersionUID 的作用**：
- 用于版本控制，反序列化时验证 UID 是否一致。
- 未显式声明时 JVM 自动计算（依赖类结构），类修改后 UID 变化导致反序列化失败。
- **最佳实践**：显式声明 `private static final long serialVersionUID = 1L;`。

## 二、Java 并发编程（10 题）

### 16. 线程的创建方式有哪些？线程的生命周期状态是怎样的？

**答案：**

**创建方式**（本质都是通过 Thread 类启动）：
1. 继承 `Thread` 类，重写 `run()` 方法。
2. 实现 `Runnable` 接口（推荐，避免单继承限制）。
3. 实现 `Callable` 接口 + `FutureTask`（可获取返回值、抛出异常）。
4. 通过线程池提交任务（生产环境推荐）。

**线程生命周期（6 种状态）**：
```
NEW（新建） -> RUNNABLE（运行/就绪） -> BLOCKED（阻塞等待锁）
                                   -> WAITING（无限期等待）
                                   -> TIMED_WAITING（有限期等待）
                                   -> TERMINATED（终止）
```

### 17. synchronized 和 ReentrantLock 的区别？

**答案：**

| 特性 | synchronized | ReentrantLock |
|------|-------------|---------------|
| **实现层面** | JVM 内置（Monitor 机制） | JDK 层面（AQS） |
| **锁释放** | 自动释放（退出代码块/异常） | 手动释放（`unlock()`，必须在 finally 中） |
| **可中断** | 不可中断 | `lockInterruptibly()` 可中断 |
| **超时获取** | 不支持 | `tryLock(timeout)` 支持 |
| **公平锁** | 非公平 | 支持公平/非公平 |
| **条件变量** | 只有一个 `wait/notify` | 支持多个 `Condition` |
| **性能** | JDK 6 后优化（偏向锁/轻量级锁），与 ReentrantLock 接近 | 略优于 synchronized（高竞争场景） |

### 18. volatile 关键字的作用是什么？它能保证原子性吗？

**答案：**

**volatile 的两大作用**：
1. **可见性**：一个线程修改 volatile 变量后，其他线程能立即看到最新值（通过内存屏障强制刷入主内存）。
2. **禁止指令重排序**：通过内存屏障（Memory Barrier）防止编译器和 CPU 重排序。

**不能保证原子性**：`volatile int count; count++;` 不是原子操作（读-改-写三步）。

**经典场景**：双重检查锁单例模式中，`volatile` 防止对象创建时指令重排序导致拿到未初始化完成的对象。

**替代方案**：需要原子性时使用 `AtomicInteger`（CAS 实现）或 `synchronized`。

### 19. 线程池的核心参数有哪些？任务提交的执行流程是怎样的？

**答案：**

**7 个核心参数**：
1. `corePoolSize`：核心线程数。
2. `maximumPoolSize`：最大线程数。
3. `keepAliveTime`：非核心线程空闲存活时间。
4. `unit`：时间单位。
5. `workQueue`：阻塞队列。
6. `threadFactory`：线程工厂。
7. `handler`：拒绝策略。

**任务提交流程**：
```
提交任务
  -> 核心线程数未满？创建核心线程执行
  -> 核心线程已满？放入阻塞队列
  -> 队列已满？创建非核心线程执行
  -> 已达最大线程数？执行拒绝策略
```

**4 种拒绝策略**：
- `AbortPolicy`（默认）：抛出 RejectedExecutionException。
- `CallerRunsPolicy`：由调用线程执行任务。
- `DiscardPolicy`：静默丢弃任务。
- `DiscardOldestPolicy`：丢弃队列头部任务，重新提交。

### 20. AQS（AbstractQueuedSynchronizer）的原理是什么？

**答案：**

AQS 是 JDK 并发包的核心框架，`ReentrantLock`、`Semaphore`、`CountDownLatch`、`ReadWriteLock` 都基于它。

**核心结构**：
- **state**：`volatile int` 状态变量，不同实现含义不同（锁计数、许可数等）。
- **CLH 队列**：FIFO 双向链表，线程获取锁失败时封装为 Node 加入队列并阻塞。
- **exclusiveOwnerThread**：独占模式下持有锁的线程。

**工作原理**：
1. 线程通过 CAS 修改 state 尝试获取锁。
2. 获取失败则封装为 Node 加入 CLH 队列尾部。
3. 队列中的线程自旋 + `LockSupport.park()` 阻塞等待。
4. 持有锁的线程释放锁后，唤醒队列中下一个节点。

**两种模式**：
- **独占模式（Exclusive）**：ReentrantLock，同一时刻只有一个线程持有。
- **共享模式（Shared）**：Semaphore、CountDownLatch，允许多个线程同时持有。

### 21. ThreadLocal 的原理是什么？为什么会导致内存泄漏？如何解决？

**答案：**

**原理**：
- 每个 Thread 内部有一个 `ThreadLocalMap`，key 是 ThreadLocal 实例（弱引用），value 是存储的值。
- 调用 `threadLocal.set(value)` 时，以当前 ThreadLocal 为 key 将值存入当前线程的 ThreadLocalMap。

**内存泄漏原因**：
- ThreadLocalMap 的 key 是弱引用，ThreadLocal 对象被 GC 回收后 key 变为 null，但 value 是强引用，不会被回收。
- 尤其在线程池场景下，线程长期存活，value 永远不会被清理。

**解决方案**：
- 使用完毕后务必调用 `threadLocal.remove()` 清理。
- 推荐使用 `private static final` 修饰 ThreadLocal，延长其生命周期。

---

### 22. CAS 是什么？有什么问题？

**答案：**

**CAS（Compare And Swap）**：一种无锁的原子操作，包含三个操作数——内存地址 V、预期值 A、新值 B。当 V 的值等于 A 时，将 V 更新为 B，否则不操作。

**CPU 指令级别**：`cmpxchg` 指令，保证原子性。

**问题**：
1. **ABA 问题**：值从 A 变为 B 再变回 A，CAS 认为没有变化。解决：`AtomicStampedReference`（加版本号）。
2. **自旋开销**：CAS 失败后自旋重试，长时间失败会浪费 CPU。
3. **只能保证单个变量的原子性**：多个变量用 `AtomicReference` 封装为一个对象或使用锁。

### 23. CountDownLatch、CyclicBarrier、Semaphore 的区别和使用场景？

**答案：**

| 工具类 | 作用 | 是否可复用 | 场景 |
|--------|------|-----------|------|
| **CountDownLatch** | 等待多个线程完成 | 不可复用 | 主线程等待 N 个子线程执行完毕 |
| **CyclicBarrier** | 多个线程互相等待到齐后一起执行 | 可复用（reset） | 多线程计算数据，最后合并结果 |
| **Semaphore** | 控制同时访问资源的线程数 | 可复用 | 限流（如数据库连接池限制 10 个并发） |

### 24. 什么是死锁？如何避免和排查？

**答案：**

**死锁的四个必要条件**：
1. 互斥条件。
2. 请求与保持条件。
3. 不剥夺条件。
4. 循环等待条件。

**避免策略**：
- 固定加锁顺序（破坏循环等待）。
- 使用 `tryLock(timeout)` 超时放弃（破坏请求与保持）。
- 减小锁的粒度，降低冲突概率。

**排查方法**：
1. `jps` 找到进程 PID。
2. `jstack <pid>` 查看线程堆栈，搜索 "Found one Java-level deadlock"。
3. 使用 `jconsole` / `jvisualvm` 的线程检测功能。
4. 生产环境可配置 `-XX:+PrintConcurrentLocks` 打印锁信息。

### 25. 虚拟线程（Virtual Thread）是什么？和传统线程有什么区别？

**答案：**

虚拟线程是 JDK 21 正式引入的轻量级线程（Project Loom）：

| 特性 | 平台线程 | 虚拟线程 |
|------|---------|---------|
| **创建成本** | 约 1MB 栈内存，与 OS 线程 1:1 映射 | 极小，与载体线程 M:N 映射 |
| **数量** | 数百到数千 | 可创建数百万 |
| **调度** | OS 内核调度 | JVM 调度（ForkJoinPool） |
| **阻塞代价** | 高（占用 OS 线程） | 低（自动 unmount，载体线程可执行其他虚拟线程） |
| **适用场景** | CPU 密集型 | I/O 密集型（网络请求、数据库操作） |

**使用方式**：
```java
Thread.startVirtualThread(() -> { /* ... */ });
// 或虚拟线程池
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> callRemoteService());
}
```

## 三、JVM（10 题）

### 26. JVM 的内存结构是怎样的？

**答案：**

JVM 运行时数据区包括：

| 区域 | 线程共享 | 作用 |
|------|---------|------|
| **程序计数器** | 私有 | 记录当前线程执行的字节码行号 |
| **虚拟机栈** | 私有 | 方法调用栈，每个方法对应一个栈帧（局部变量表、操作数栈、动态链接、返回地址） |
| **本地方法栈** | 私有 | Native 方法调用 |
| **堆（Heap）** | 共享 | 对象实例、数组，GC 的主要区域 |
| **方法区（Metaspace，JDK 8+）** | 共享 | 类元信息、常量池、静态变量（使用本地内存） |

**堆的分代结构**：
- **新生代**：Eden 区（80%）+ Survivor 0/1 区（各 10%），使用复制算法。
- **老年代**：存活时间长的对象，使用标记-整理算法。

### 27. 对象在 JVM 中的创建过程是怎样的？

**答案：**

1. **类加载检查**：遇到 new 指令时，检查类是否已加载、链接、初始化。
2. **分配内存**：
   - 内存规整（Serial、ParNew）：指针碰撞（Bump the Pointer）。
   - 内存不规整（CMS）：空闲列表（Free List）。
   - 并发分配：CAS + 失败重试 或 TLAB（Thread Local Allocation Buffer）。
3. **内存空间初始化为零值**：保证不赋初值也能使用默认值。
4. **设置对象头**：Mark Word（哈希码、GC 分代年龄、锁标志位）、类型指针、数组长度。
5. **执行 `<init>` 方法**：构造器，按程序代码赋值。

### 28. 如何判断一个对象是否可以被回收？

**答案：**

**1. 引用计数法**（Python 使用，JVM 不使用）：
- 对象被引用时计数 +1，引用失效时 -1，为 0 时回收。
- 问题：无法解决循环引用。

**2. 可达性分析算法**（JVM 使用）：
- 从 GC Roots 出发，能到达的对象都是存活的，不可达的对象可回收。
- **GC Roots 包括**：
  - 虚拟机栈中引用的对象。
  - 方法区中静态变量引用的对象。
  - 方法区中常量引用的对象。
  - JNI（Native 方法）引用的对象。

**四种引用类型**：强引用（永不回收）> 软引用（内存不足时回收）> 弱引用（下次 GC 回收）> 虚引用（随时回收，用于跟踪 GC 活动）。

### 29. 常见的垃圾回收算法有哪些？

**答案：**

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **标记-清除** | 标记可回收对象后直接清除 | 简单 | 内存碎片 |
| **标记-复制** | 将存活对象复制到另一块区域 | 无碎片、分配快 | 浪费一半内存 |
| **标记-整理** | 标记后将存活对象向一端移动 | 无碎片 | 移动开销大 |
| **分代收集** | 新生代用复制，老年代用标记-整理 | 综合最优 | 实现复杂 |

### 30. 常见的垃圾回收器有哪些？G1 和 ZGC 有什么区别？

**答案：**

| 回收器 | 分代 | 算法 | 特点 |
|--------|------|------|------|
| **Serial** | 新生代 | 复制 | 单线程，STW，Client 模式 |
| **ParNew** | 新生代 | 复制 | Serial 的多线程版 |
| **Parallel Scavenge** | 新生代 | 复制 | 关注吞吐量 |
| **CMS** | 老年代 | 标记-清除 | 低延迟，有碎片，JDK 14 移除 |
| **G1** | 整堆 | Region + 复制/整理 | JDK 9 默认，可预测停顿时间 |
| **ZGC** | 整堆 | 着色指针 + 读屏障 | 亚毫秒级停顿，JDK 21 正式生产可用 |

**G1 vs ZGC**：
- **G1**：将堆划分为多个 Region，混合回收，停顿时间可控（`-XX:MaxGCPauseMillis`），适合 8G 以下堆。
- **ZGC**：基于着色指针（Colored Pointers），并发标记和并发转移，停顿时间 < 1ms，支持 TB 级堆内存，JDK 21 的分代 ZGC 性能更优。

### 31. 什么是类加载机制？双亲委派模型是什么？

**答案：**

**类加载过程**：加载 -> 链接（验证 -> 准备 -> 解析）-> 初始化。

**双亲委派模型**：
```
Bootstrap ClassLoader（启动类加载器：rt.jar）
    |
Extension ClassLoader（扩展类加载器：ext/*.jar）
    |
Application ClassLoader（应用类加载器：classpath）
    |
Custom ClassLoader（自定义类加载器）
```

**工作原理**：收到加载请求时，先委托父加载器加载，父加载器无法加载时才自己加载。

**好处**：
- 避免类的重复加载。
- 保护核心 API 不被篡改（安全沙箱）。

**打破双亲委派**：
- SPI 机制（JDBC 驱动）：使用线程上下文类加载器（Thread Context ClassLoader）。
- OSGi、Tomcat：自定义类加载器优先加载自己的类。
- 热部署：自定义 ClassLoader 加载新版本类。

### 33. 线上 OOM（OutOfMemoryError）如何排查？

**答案：**

**排查步骤**：
1. **保留现场**：确保配置了 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/dump.hprof`。
2. **分析 Heap Dump**：
   - 使用 MAT（Memory Analyzer Tool）或 JProfiler 打开 dump 文件。
   - 查看 "Leak Suspects Report"，找出大对象和 GC Roots 引用链。
3. **常见 OOM 类型**：
   - `Java heap space`：堆内存不足，检查是否有大对象或内存泄漏。
   - `Metaspace`：类加载过多（动态代理、热部署）。
   - `GC overhead limit exceeded`：GC 消耗 98% 以上 CPU 且回收不到 2% 内存。
   - `Direct buffer memory`：NIO 直接内存溢出。
   - `Unable to create new native thread`：线程数超过 OS 限制。
4. **定位代码**：通过引用链追溯到具体的代码位置，修复内存泄漏或调整内存参数。

### 34. 什么是 JIT 编译？C1 和 C2 编译器有什么区别？

**答案：**

**JIT（Just-In-Time）编译**：JVM 在运行时将热点代码（方法或循环体）从字节码编译为本地机器码，提升执行效率。

| 特性 | C1（Client） | C2（Server） |
|------|-------------|-------------|
| **优化程度** | 轻量优化 | 深度优化（逃逸分析、标量替换、锁消除等） |
| **编译速度** | 快 | 慢 |
| **生成代码质量** | 一般 | 高 |
| **适用场景** | 启动阶段 | 长期运行的热点代码 |

**分层编译（默认开启）**：
- Level 0：解释执行。
- Level 1-3：C1 编译，逐步加入 profiling 数据收集。
- Level 4：C2 编译，使用 profiling 数据进行激进优化。
- JDK 17+ 的 GraalVM 提供 Graal 编译器替代 C2，进一步优化。

### 35. 什么是逃逸分析？有哪些优化手段？

**答案：**

逃逸分析判断对象的作用域是否超出当前方法：

- **方法逃逸**：对象被返回或赋值给全局变量，可能被外部方法访问。
- **线程逃逸**：对象被传递给其他线程。

**基于逃逸分析的优化**：
1. **栈上分配**：不逃逸的对象直接在栈上分配内存，方法结束自动回收，减轻堆 GC 压力。
2. **标量替换**：将对象拆解为基本类型（标量），直接在栈上分配，避免对象创建开销。
3. **锁消除**：对不会逃逸到多线程的对象，消除同步锁（如 StringBuffer 在单线程中自动去锁）。

## 四、MySQL 数据库（10 题）

### 36. MySQL 的索引数据结构是什么？为什么使用 B+ 树而不是 B 树？

**答案：**

MySQL InnoDB 使用 **B+ 树** 作为索引数据结构。

**B+ 树 vs B 树**：

| 特性 | B 树 | B+ 树 |
|------|------|------|
| **数据存储** | 每个节点存储数据 | 只有叶子节点存储数据 |
| **叶子节点链表** | 无 | 叶子节点通过双向链表连接 |
| **查询稳定性** | 查询路径长度不一 | 所有查询都到叶子节点，性能稳定 |
| **范围查询** | 需中序遍历 | 沿叶子链表顺序扫描，高效 |
| **磁盘 I/O** | 非叶子节点占空间，树更高 | 非叶子节点只存 key，树更矮，I/O 更少 |

**聚簇索引 vs 非聚簇索引**：
- **聚簇索引**：主键索引，叶子节点存储完整行数据，一张表只有一个。
- **非聚簇索引（二级索引）**：叶子节点存储主键值，需要**回表**查询完整数据。

### 37. 什么是覆盖索引和索引下推？

**答案：**

**覆盖索引**：查询的字段全部包含在索引中，无需回表。`EXPLAIN` 中 `Extra` 列显示 `Using index`。

```sql
-- 联合索引 (name, age)
SELECT name, age FROM user WHERE name = 'Tom'; -- 覆盖索引
SELECT * FROM user WHERE name = 'Tom';          -- 需要回表
```

**索引下推（Index Condition Pushdown, ICP）**：MySQL 5.6 引入，在索引遍历过程中过滤不满足条件的记录，减少回表次数。`EXPLAIN` 中 `Extra` 显示 `Using index condition`。

```sql
-- 联合索引 (name, age)
SELECT * FROM user WHERE name LIKE '张%' AND age = 10;
-- 没有 ICP：先通过 name 索引找到所有"张"开头的记录，逐一回表后过滤 age
-- 有 ICP：在索引层面同时判断 age = 10，只回表满足条件的记录
```

### 38. 哪些情况会导致索引失效？

**答案：**

1. **对索引列使用函数或运算**：`WHERE YEAR(create_time) = 2025`。
2. **隐式类型转换**：`WHERE phone = 13800138000`（phone 是 varchar，传入数字）。
3. **LIKE 以通配符开头**：`WHERE name LIKE '%张'`。
4. **OR 条件中有非索引列**：`WHERE indexed_col = 1 OR non_indexed_col = 2`。
5. **NOT IN / NOT EXISTS**：某些情况下优化器放弃使用索引。
6. **联合索引不满足最左前缀原则**：索引 (a, b, c)，查询条件只有 b、c。
7. **索引列参与计算**：`WHERE a + 1 = 5`。
8. **字符集不匹配**：关联表的字符集或排序规则不一致。

**排查方式**：使用 `EXPLAIN` 查看执行计划，关注 `type`（至少 `ref` 或 `range`）、`key`、`rows`、`Extra`。

### 39. MySQL 的事务隔离级别有哪些？InnoDB 默认是什么？

**答案：**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|----------|------|
| READ UNCOMMITTED | 可能 | 可能 | 可能 |
| READ COMMITTED | 不可能 | 可能 | 可能 |
| **REPEATABLE READ（InnoDB 默认）** | 不可能 | 不可能 | InnoDB 通过 MVCC + 间隙锁基本解决 |
| SERIALIZABLE | 不可能 | 不可能 | 不可能 |

**InnoDB 的 REPEATABLE READ 如何解决幻读**：
- **快照读**（普通 SELECT）：通过 MVCC 保证读取的是事务开始时的快照版本。
- **当前读**（SELECT FOR UPDATE / INSERT / UPDATE / DELETE）：通过 Next-Key Lock（行锁 + 间隙锁）锁住范围，防止其他事务插入。

### 40. MVCC 的原理是什么？

**答案：**

**MVCC（Multi-Version Concurrency Control）** 通过为每行数据维护多个版本实现非锁定的一致性读。

**核心组件**：
1. **隐藏字段**：
   - `DB_TRX_ID`：最近修改该行的事务 ID。
   - `DB_ROLL_PTR`：回滚指针，指向 undo log 中的旧版本。
2. **Undo Log 版本链**：通过 ROLL_PTR 将数据的历史版本串成链表。
3. **ReadView（读视图）**：
   - `m_ids`：生成 ReadView 时活跃的事务 ID 列表。
   - `min_trx_id` / `max_trx_id`：活跃事务 ID 范围。
   - `creator_trx_id`：创建该 ReadView 的事务 ID。

**可见性判断**：
- 行的 `DB_TRX_ID < min_trx_id`：该行在创建 ReadView 前已提交，可见。
- 行的 `DB_TRX_ID >= max_trx_id`：该行在创建 ReadView 后才开始，不可见。
- 行的 `DB_TRX_ID` 在 `m_ids` 中：说明该事务在创建 ReadView 时还活跃，不可见。

**RC vs RR**：RC 每次 SELECT 都生成新 ReadView，RR 只在第一次 SELECT 时生成。

### 41. MySQL 的日志系统有哪些？各自的作用是什么？

**答案：**

| 日志 | 层级 | 作用 |
|------|------|------|
| **redo log** | InnoDB | 物理日志，记录数据页的修改，用于崩溃恢复（Crash-Safe） |
| **undo log** | InnoDB | 逻辑日志，记录数据的反向操作，用于事务回滚和 MVCC |
| **binlog** | MySQL Server | 逻辑日志，记录所有变更语句，用于主从复制和数据恢复 |
| **slow query log** | MySQL Server | 记录执行时间超过阈值的 SQL |
| **error log** | MySQL Server | 记录启动、关闭和错误信息 |
| **general log** | MySQL Server | 记录所有 SQL（通常关闭） |

**两阶段提交（保证 redo log 和 binlog 一致）**：
1. Prepare 阶段：写 redo log，标记 prepare 状态。
2. 写 binlog。
3. Commit 阶段：写 redo log，标记 commit 状态。
4. 崩溃恢复时：redo log prepare + binlog 完整 -> 提交；否则回滚。

### 42. 如何优化慢查询？

**答案：**

**排查步骤**：
1. **开启慢查询日志**：`slow_query_log = ON`，`long_query_time = 1`。
2. **使用 EXPLAIN 分析执行计划**：关注 type（ALL -> index -> range -> ref -> eq_ref -> const）、key、rows、Extra。
3. **使用 SHOW PROFILE 分析 SQL 各阶段耗时**。

**常见优化手段**：
- **索引优化**：添加合适的索引、避免索引失效、使用覆盖索引。
- **SQL 改写**：避免 `SELECT *`、减少子查询（改用 JOIN）、分页优化（延迟关联）。
- **深分页优化**：
  ```sql
  -- 慢：SELECT * FROM orders LIMIT 1000000, 10;
  -- 快：SELECT * FROM orders WHERE id > (SELECT id FROM orders LIMIT 1000000, 1) LIMIT 10;
  ```
- **表结构优化**：垂直拆分（大字段分表）、水平分表（数据量大时）。
- **参数调优**：`innodb_buffer_pool_size`、`sort_buffer_size`、`join_buffer_size`。

### 43. MySQL 主从复制的原理是什么？如何解决主从延迟？

**答案：**

**复制流程**：
1. **Master**：将变更写入 binlog。
2. **Slave I/O Thread**：从 Master 拉取 binlog，写入本地 relay log。
3. **Slave SQL Thread**：读取 relay log 并重放，应用到从库。

**复制模式**：
- **异步复制**（默认）：Master 不等 Slave 确认。
- **半同步复制**：Master 等至少一个 Slave 确认收到 binlog。
- **全同步复制**（MGR）：所有 Slave 都确认。

**主从延迟解决方案**：
1. **强制走主库**：写操作后立即读取时走主库。
2. **延迟敏感业务读主库**：核心业务直接读主库。
3. **并行复制**：MySQL 5.7+ 基于 `LOGICAL_CLOCK` 的并行复制，Slave 多线程重放。
4. **GTID 模式**：简化主从切换和故障恢复。

### 44. 分库分表方案有哪些？什么时候需要分库分表？

**答案：**

**何时分库分表**：
- 单表数据量超过 **500 万 ~ 1000 万** 行（或单表容量超过 2GB）。
- 数据库整体 QPS 超过单库承载能力。

**方案**：
| 方案 | 方式 | 适用场景 |
|------|------|---------|
| **垂直分表** | 将大字段或不常用字段拆分到扩展表 | 表中有 text/blob 大字段 |
| **水平分表** | 按规则（hash/range）将数据分散到多个表 | 单表数据量过大 |
| **垂直分库** | 按业务模块拆分到不同数据库 | 不同业务耦合在同一库 |
| **水平分库** | 将同一张表的数据分散到多个库 | 单库写入瓶颈 |

**常见问题**：
- **分布式 ID**：雪花算法（Snowflake）、UUID、号段模式。
- **跨库 JOIN**：冗余字段、应用层组装、宽表。
- **分页查询**：二次排序归并。
- **常用中间件**：ShardingSphere、MyCat。

### 45. MySQL 的死锁是如何产生的？如何排查和解决？

**答案：**

**产生条件**：两个或多个事务互相持有对方需要的锁，形成循环等待。

**示例**：
```
事务 A：UPDATE t SET ... WHERE id = 1; -- 持有 id=1 的行锁
事务 B：UPDATE t SET ... WHERE id = 2; -- 持有 id=2 的行锁
事务 A：UPDATE t SET ... WHERE id = 2; -- 等待 id=2 的锁
事务 B：UPDATE t SET ... WHERE id = 1; -- 等待 id=1 的锁 -> 死锁
```

**排查方法**：
1. `SHOW ENGINE INNODB STATUS\G`：查看 `LATEST DETECTED DEADLOCK` 部分。
2. 开启死锁日志：`innodb_print_all_deadlocks = ON`。
3. 分析 `information_schema.INNODB_LOCKS` 和 `INNODB_LOCK_WAITS`。

**解决方案**：
- 事务中按固定顺序访问表和行。
- 缩短事务，减小锁持有时间。
- 降低隔离级别为 READ COMMITTED（无间隙锁）。
- 为查询添加合适的索引，避免全表扫描锁表。

## 五、Redis（10 题）

### 46. Redis 为什么这么快？

**答案：**

1. **纯内存操作**：数据存储在内存中，读写速度极快（纳秒级）。
2. **单线程模型**（核心处理）：避免多线程上下文切换和锁竞争的开销。注意：Redis 6.0 引入多线程处理网络 I/O（io-threads），但命令执行仍是单线程。
3. **高效数据结构**：跳表（Skip List）、压缩列表（Ziplist）、整数集合（Intset）、SDS 等专门为性能优化的数据结构。
4. **I/O 多路复用**：使用 epoll/kqueue 处理大量并发连接。
5. **协议简单高效**：RESP 协议解析速度快。

---

### 47. Redis 的持久化机制有哪些？RDB 和 AOF 的区别？

**答案：**

| 特性 | RDB（快照） | AOF（追加日志） |
|------|-----------|----------------|
| **原理** | 某一时刻的全量内存快照 | 记录每一条写命令 |
| **文件大小** | 小（二进制压缩） | 大（文本命令） |
| **恢复速度** | 快 | 慢（需重放命令） |
| **数据安全** | 可能丢失最后一次快照后的数据 | 取决于 fsync 策略 |
| **性能影响** | fork 子进程时有短暂阻塞 | everysec 几乎无影响 |

**AOF 的 fsync 策略**：
- `always`：每条命令都 fsync，最安全但最慢。
- `everysec`（默认）：每秒 fsync，最多丢 1 秒数据。
- `no`：由 OS 决定 fsync。

**Redis 4.0+ 混合持久化**：AOF 文件前半段是 RDB 格式（全量），后半段是 AOF 增量命令，兼顾恢复速度和数据安全。

---

### 48. Redis 的过期策略和内存淘汰策略是什么？

**答案：**

**过期删除策略**：
1. **惰性删除**：访问 key 时检查是否过期，过期则删除。
2. **定期删除**：每 100ms 随机检查一批有过期时间的 key，删除已过期的。

**内存淘汰策略**（内存满时 `maxmemory-policy`）：
| 策略 | 说明 |
|------|------|
| `noeviction`（默认） | 不淘汰，写入返回错误 |
| `allkeys-lru` | 所有 key 中淘汰最近最少使用的（最常用） |
| `volatile-lru` | 有过期时间的 key 中淘汰 LRU |
| `allkeys-lfu` | 所有 key 中淘汰最不常用的（Redis 4.0+） |
| `volatile-lfu` | 有过期时间的 key 中淘汰 LFU |
| `allkeys-random` | 随机淘汰 |
| `volatile-random` | 有过期时间的 key 中随机淘汰 |
| `volatile-ttl` | 淘汰剩余 TTL 最短的 |

---

### 49. 缓存穿透、缓存击穿、缓存雪崩分别是什么？如何解决？

**答案：**

| 问题 | 定义 | 解决方案 |
|------|------|---------|
| **缓存穿透** | 查询不存在的数据，请求直接打到数据库 | 1. 布隆过滤器拦截 2. 缓存空值（短过期时间） |
| **缓存击穿** | 某个热点 key 过期，大量请求同时打到数据库 | 1. 互斥锁（SETNX）2. 逻辑过期（不设 TTL，异步更新）3. 热点 key 永不过期 |
| **缓存雪崩** | 大批量 key 同时过期或 Redis 宕机 | 1. 过期时间加随机值 2. 多级缓存 3. 限流降级 4. Redis 高可用（集群/哨兵） |

---

### 50. 如何保证缓存和数据库的一致性？

**答案：**

**常用策略**：**Cache Aside Pattern（旁路缓存）**

1. **读操作**：先读缓存 -> 缓存未命中 -> 读数据库 -> 写入缓存。
2. **写操作**：**先更新数据库，再删除缓存**（推荐）。

**为什么是删除缓存而不是更新缓存**：
- 更新缓存在并发写时可能出现数据不一致。
- 写多读少的场景，频繁更新缓存是浪费。

**保证一致性的进阶方案**：
- **延迟双删**：先删缓存 -> 更新数据库 -> 延迟一段时间再删缓存（解决读写并发问题）。
- **消息队列重试**：删除缓存失败时，将删除操作放入 MQ 重试。
- **Canal 监听 binlog**：通过订阅 MySQL binlog 异步更新/删除缓存，解耦业务代码。

---

### 51. Redis 的集群方案有哪些？

**答案：**

| 方案 | 原理 | 特点 |
|------|------|------|
| **主从复制** | 主节点写，从节点读 | 读写分离，无高可用 |
| **哨兵模式（Sentinel）** | 哨兵监控主从节点，主节点故障自动切换 | 高可用，但写能力受限于单节点 |
| **Cluster 集群** | 16384 个 hash slot 分配到多个主节点 | 水平扩展，去中心化，自带故障转移 |

**Cluster 原理**：
- 每个 key 通过 `CRC16(key) % 16384` 计算所属 slot。
- 每个主节点负责一部分 slot，每个主节点有从节点做冗余。
- 节点间通过 Gossip 协议通信，超过半数主节点认为某节点不可达时标记 `fail`。

---

### 52. Redis 分布式锁如何实现？有什么注意事项？

**答案：**

**基本实现**：
```
SET key value NX PX 30000  // 原子性设置锁和过期时间
```
释放锁：Lua 脚本保证 get + compare + delete 的原子性。

**注意事项**：
1. **锁过期但业务未完成**：使用 Redisson 的看门狗（WatchDog）机制自动续期。
2. **Redis 主从切换导致锁丢失**：主节点加锁后未同步到从节点就宕机，从节点升级后锁丢失。解决方案：**RedLock 算法**（向 N 个独立 Redis 节点加锁，超过半数成功才算获取锁），但存在争议。
3. **可重入锁**：Redisson 通过 hash 结构记录锁持有者和重入次数。
4. **公平锁/联锁/读写锁**：Redisson 均提供实现。

### 53. Redis 的大 Key 问题如何处理？

**答案：**

**什么是大 Key**：
- String 类型 > 10KB。
- 集合类型元素超过 5000 个。

**危害**：
- 阻塞其他请求（单线程）。
- 内存不均匀。
- 集群模式下数据迁移困难。

**处理方式**：
1. **发现大 Key**：`redis-cli --bigkeys`、`MEMORY USAGE key`、RDB 分析工具（redis-rdb-tools）。
2. **拆分**：
   - Hash 分片：一个大 Hash 拆为多个小 Hash（如 `user:info:{uid}` -> `user:info:{uid}:{shard}`）。
   - 集合分片：大 List/Set/ZSet 拆为多个小集合。
3. **异步删除**：使用 `UNLINK`（Redis 4.0+）代替 `DEL`，后台线程异步释放内存。
4. **预防**：开发规范限制 value 大小，CI 中集成大 Key 检测。

---

### 54. Redis 的热 Key 问题如何处理？

**答案：**

**什么是热 Key**：某个 key 的访问 QPS 极高（如秒杀商品库存）。

**危害**：单节点压力过大，可能导致该节点 CPU 打满。

**发现方式**：
1. `redis-cli --hotkeys`（需开启 LFU 淘汰策略）。
2. 业务层面监控统计。
3. Redis 的 `MONITOR` 命令（生产慎用，影响性能）。

**处理方式**：
1. **本地缓存**：在应用层使用 Caffeine/Guava Cache 缓存热 Key 数据，减少 Redis 请求。
2. **读写分离 + 多从节点**：将热 Key 的读请求分散到多个从节点。
3. **Key 加随机后缀**：`hotkey:random(1~N)`，分散到不同 slot/节点。
4. **热点感知**：京东的 hotkey 框架，实时发现热 Key 并自动推送到本地缓存。

---

### 55. Redis 7.0 有哪些重要新特性？

**答案：**

1. **Redis Functions**：取代 `EVAL`，支持持久化 Lua 函数，可像命令一样调用。
2. **Multi-part AOF**：将 AOF 拆分为多个文件（base + incr），提升 AOF 重写性能。
3. **Client-side caching 优化**：改进客户端缓存协议（Tracking 模式）。
4. **Sharded Pub/Sub**：Pub/Sub 消息按 channel 分片到集群节点，避免全节点广播。
5. **ACL v2**：更细粒度的权限控制，支持 selector。
6. **Listpack 替代 Ziplist**：解决 Ziplist 的连锁更新问题。
7. **Memory efficiency**：优化小对象的内存占用。

## 六、Spring / Spring Boot（10 题）

### 56. Spring IoC 容器的原理是什么？

**答案：**

**IoC（控制反转）**：将对象的创建和依赖管理从代码中转移到容器。

**核心流程**：
1. **Bean 定义读取**：通过 `@Configuration` + `@Bean`、`@Component`、XML 等方式定义 Bean。
2. **BeanDefinition 注册**：解析为 `BeanDefinition` 对象，注册到 `BeanDefinitionRegistry`。
3. **Bean 创建**：
   - 实例化（Instantiation）：反射创建对象。
   - 属性填充（Populate）：注入依赖（`@Autowired`、`@Value`）。
   - 初始化（Initialization）：执行 `Aware` 接口、`BeanPostProcessor`、`@PostConstruct`、`InitializingBean`。
4. **Bean 使用**：从容器获取 Bean。
5. **Bean 销毁**：执行 `@PreDestroy`、`DisposableBean`。

**三级缓存解决循环依赖**：
- 一级缓存 `singletonObjects`：完全初始化的 Bean。
- 二级缓存 `earlySingletonObjects`：提前暴露的半成品 Bean。
- 三级缓存 `singletonFactories`：ObjectFactory，用于生成早期引用（支持 AOP 代理）。

---

### 57. Spring AOP 的原理和实现方式是什么？

**答案：**

**AOP（面向切面编程）**：将横切关注点（日志、事务、权限）从业务逻辑中分离。

**实现方式**：
- **JDK 动态代理**：目标类实现了接口时，通过 `Proxy` 生成接口代理类。
- **CGLIB**：目标类没有实现接口时，生成目标类的子类作为代理（不能代理 final 类/方法）。

**核心概念**：
- **切面（Aspect）**：横切逻辑的模块化（`@Aspect`）。
- **切入点（Pointcut）**：定义哪些方法被拦截。
- **通知（Advice）**：`@Before`、`@After`、`@Around`、`@AfterReturning`、`@AfterThrowing`。
- **连接点（JoinPoint）**：程序执行中的某个点（方法调用、异常抛出等）。

---

### 58. Spring Boot 的自动装配原理是什么？

**答案：**

**核心注解**：`@SpringBootApplication` = `@Configuration` + `@EnableAutoConfiguration` + `@ComponentScan`。

**自动装配流程**：
1. `@EnableAutoConfiguration` 通过 `@Import(AutoConfigurationImportSelector.class)` 导入自动配置选择器。
2. `AutoConfigurationImportSelector` 读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件（Spring Boot 3.x）或 `META-INF/spring.factories`（Spring Boot 2.x）。
3. 获取所有自动配置类的全限定名。
4. 通过 `@Conditional` 系列注解（`@ConditionalOnClass`、`@ConditionalOnMissingBean`、`@ConditionalOnProperty` 等）判断是否满足条件。
5. 满足条件的配置类被加载，自动创建 Bean 并注入到容器。

**自定义 Starter**：
1. 创建 `autoconfigure` 模块，定义配置类和 `@ConfigurationProperties`。
2. 创建 `starter` 模块，依赖 `autoconfigure` 模块。
3. 在 `META-INF/spring/` 下注册自动配置类。

### 59. Spring 的事务传播机制有哪些？

**答案：**

| 传播行为 | 说明 |
|---------|------|
| **REQUIRED（默认）** | 有事务则加入，无事务则新建 |
| **SUPPORTS** | 有事务则加入，无事务则非事务执行 |
| **MANDATORY** | 有事务则加入，无事务则抛异常 |
| **REQUIRES_NEW** | 总是新建事务，如果存在事务则挂起当前事务 |
| **NOT_SUPPORTED** | 总是非事务执行，如果存在事务则挂起 |
| **NEVER** | 总是非事务执行，如果存在事务则抛异常 |
| **NESTED** | 如果存在事务则在嵌套事务中执行（savepoint），无事务等同 REQUIRED |

**注意**：`@Transactional` 基于 AOP 代理实现，同类内部方法调用（this 调用）事务不生效。解决方案：注入自身、使用 `AopContext.currentProxy()`、拆分类。

### 60. Spring Bean 的作用域有哪些？

**答案：**

| 作用域 | 说明 |
|--------|------|
| **singleton（默认）** | 整个 IoC 容器只有一个实例 |
| **prototype** | 每次请求都创建新实例 |
| **request** | 每个 HTTP 请求一个实例（Web 环境） |
| **session** | 每个 HTTP Session 一个实例 |
| **application** | 每个 ServletContext 一个实例 |
| **websocket** | 每个 WebSocket 会话一个实例 |

**注意**：singleton Bean 注入 prototype Bean 时，prototype 不会每次创建新实例。解决方案：`@Lookup` 注解、`ObjectFactory<T>`、`Provider<T>`。

### 61. @Autowired 和 @Resource 的区别是什么？

**答案：**

| 特性 | @Autowired | @Resource |
|------|-----------|-----------|
| **来源** | Spring 框架（`org.springframework`） | JDK（`javax.annotation` / `jakarta.annotation`） |
| **注入方式** | 默认按类型（byType），可配合 `@Qualifier` 按名称 | 默认按名称（byName），找不到再按类型 |
| **指定名称** | `@Qualifier("beanName")` | `@Resource(name = "beanName")` |
| **可用位置** | 构造器、字段、setter | 字段、setter（不支持构造器） |

**推荐做法**：使用构造器注入（Spring 官方推荐），保证不可变性和必要依赖。

### 62. Spring Boot 3.x 相比 2.x 有哪些重大变化？

**答案：**

1. **Java 17 最低要求**：Spring Boot 3.0 要求 Java 17+。
2. **Jakarta EE 迁移**：`javax.*` 包名全部替换为 `jakarta.*`（如 `javax.servlet` -> `jakarta.servlet`）。
3. **Spring Security 6**：安全配置方式大幅变化，弃用 `WebSecurityConfigurerAdapter`。
4. **GraalVM 原生镜像支持**：支持将应用编译为 Native Image，启动时间 < 0.1s，内存占用大幅降低。
5. **可观测性增强**：Micrometer Observation API 统一 Metrics 和 Tracing。
6. **HTTP Interface**：类似 Feign 的声明式 HTTP 客户端，但更轻量。
7. **自动装配注册文件变更**：`spring.factories` -> `AutoConfiguration.imports`。

### 63. Spring 中的事件机制（Event）是怎样的？

**答案：**

Spring 事件机制基于观察者模式：

**核心组件**：
- **事件（ApplicationEvent）**：`ContextRefreshedEvent`、`ContextStartedEvent`、`RequestHandledEvent` 等。
- **发布者（ApplicationEventPublisher）**：ApplicationContext 实现了该接口。
- **监听器（ApplicationListener）**：实现 `ApplicationListener<T>` 接口或使用 `@EventListener` 注解。

**使用示例**：
```java
@EventListener
public void onOrderCreated(OrderCreatedEvent event) {
    // 处理订单创建事件
}

@Async
@EventListener
public void onOrderCreatedAsync(OrderCreatedEvent event) {
    // 异步处理
}
```

**应用场景**：业务解耦（如下单后发短信、加积分）、启动完成后执行初始化任务。

### 64. Spring Boot 的 Actuator 是什么？生产环境如何使用？

Actuator 提供生产级别的监控和管理端点：

**常用端点**：

### 65. 如何在 Spring Boot 中实现优雅停机？

**答案：**

**Spring Boot 2.3+ 内置优雅停机支持**：

```yaml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

**工作流程**：
1. 收到 SIGTERM 信号。
2. 健康检查返回 OUT_OF_SERVICE（从注册中心下线）。
3. 停止接收新请求。
4. 等待正在处理的请求完成（最多等待 `timeout-per-shutdown-phase`）。
5. 关闭线程池、数据库连接池等资源。
6. 发布 `ContextClosedEvent`。

**注意事项**：
- K8s 的 `preStop` hook 配合使用。
- 消息消费者需要实现 `SmartLifecycle` 接口，在 `stop()` 中停止消费并等待处理完成。
- 定时任务使用 `@SchedulerLock` 避免在停机过程中执行。

### 67. Nacos 作为注册中心，AP 和 CP 模式有什么区别？

**答案：**

| 特性 | AP 模式（默认，Distro 协议） | CP 模式（Raft 协议） |
|------|---------------------------|---------------------|
| **一致性** | 最终一致性 | 强一致性 |
| **可用性** | 高（节点故障不影响服务发现） | 低（选举期间不可用） |
| **实例类型** | 临时实例（ephemeral = true） | 持久化实例 |
| **适用场景** | 大多数微服务场景 | 需要强一致的服务（如 DNS） |

**切换方式**：
```yaml
spring:
  cloud:
    nacos:
      discovery:
        ephemeral: false  # 切换为 CP 模式
```

**Nacos 与 Eureka 的区别**：
- Nacos 支持 AP/CP 切换，Eureka 只有 AP。
- Nacos 支持服务权重、元数据、配置管理。
- Nacos 支持服务端主动推送变更，Eureka 依赖客户端轮询。

### 68. 微服务之间的通信方式有哪些？各有什么优缺点？

**答案：**

| 方式 | 类型 | 优点 | 缺点 |
|------|------|------|------|
| **REST（HTTP）** | 同步 | 简单、通用、易调试 | 耦合度高、延迟累积 |
| **OpenFeign** | 同步 | 声明式、易用 | 本质是 REST |
| **gRPC** | 同步 | 高性能（Protobuf）、强类型 | 调试不友好 |
| **消息队列（Kafka/RocketMQ）** | 异步 | 解耦、削峰、可靠 | 最终一致性、延迟 |
| **事件驱动** | 异步 | 高度解耦 | 复杂度高 |

**选型建议**：
- 查询类用同步（REST/gRPC）。
- 通知类、耗时操作用异步（MQ）。
- 高性能内部通信用 gRPC。
- 需要可靠传递用 MQ + 事务消息。

### 69. 服务熔断、降级、限流的区别和实现方式？

**答案：**

| 策略 | 目的 | 实现方式 |
|------|------|---------|
| **熔断** | 下游故障时快速失败，防止级联故障 | 状态机（Closed -> Open -> Half-Open），错误率达到阈值时熔断 |
| **降级** | 服务不可用时返回兜底数据 | fallback 方法，返回默认值或缓存数据 |
| **限流** | 控制流量，保护系统 | 固定窗口、滑动窗口、漏桶、令牌桶算法 |

**Sentinel vs Hystrix**：

| 特性 | Sentinel | Hystrix |
|------|---------|---------|
| **隔离策略** | 基于并发线程数或信号量 | 线程池/信号量 |
| **熔断策略** | 基于响应时间或异常比率 | 基于异常比率 |
| **限流** | 支持（多种流控规则） | 有限支持 |
| **实时指标** | 滑动窗口 | 滑动窗口 |
| **控制台** | 有（Sentinel Dashboard） | 无（需集成） |
| **维护状态** | 活跃 | 停止更新 |

### 70. Spring Cloud Gateway 的核心概念和工作流程？

**答案：**

**核心概念**：
- **Route（路由）**：定义请求到哪个微服务，由断言和过滤器组成。
- **Predicate（断言）**：匹配条件（路径、Header、Method 等）。
- **Filter（过滤器）**：请求/响应处理逻辑（限流、鉴权、日志、跨域等）。

**工作流程**：
```
客户端请求
  -> Gateway Handler Mapping（路由匹配）
  -> Gateway Web Handler（过滤器链）
  -> Pre Filters（前置处理：鉴权、限流、参数改写）
  -> 下游微服务
  -> Post Filters（后置处理：添加 Header、日志记录）
  -> 返回响应
```

**常用功能**：
- 统一鉴权（JWT 验证）。
- 动态路由（从 Nacos 读取路由配置）。
- 限流（`RequestRateLimiter` 基于 Redis + Lua）。
- 灰度发布（按权重/Header 路由）。

### 71. 分布式事务有哪些解决方案？Seata 的模式有哪些？

**答案：**

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|---------|
| **2PC（XA）** | 强一致 | 低 | 低 | 金融核心系统 |
| **TCC（Try-Confirm-Cancel）** | 最终一致 | 高 | 高（需写 3 个接口） | 资金类业务 |
| **SAGA** | 最终一致 | 高 | 中 | 长事务 |
| **本地消息表** | 最终一致 | 高 | 中 | 通用 |
| **事务消息（RocketMQ）** | 最终一致 | 高 | 中 | RocketMQ 用户 |

**Seata 四种模式**：
1. **AT 模式**（默认）：基于本地事务 + 全局锁，自动生成回滚 SQL（undo_log），无侵入，适合大多数场景。
2. **TCC 模式**：手动编写 Try/Confirm/Cancel，性能好，适合资金类高并发场景。
3. **SAGA 模式**：长事务拆分为多个本地事务 + 补偿操作。
4. **XA 模式**：标准 2PC，强一致但性能低。

### 74. 微服务如何保证接口幂等性？

**答案：**

**实现方案**：

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| **唯一请求 ID** | 每个请求携带唯一 ID，服务端去重 | 通用 |
| **数据库唯一约束** | INSERT 时利用唯一索引防重 | 创建类操作 |
| **乐观锁（版本号）** | UPDATE 时带版本号，更新后版本号 +1 | 更新类操作 |

| **Token 机制** | 先获取 Token，请求携带 Token，服务端验证并删除 | 表单重复提交 |
| **SELECT + INSERT/UPDATE** | 先查询是否已存在，再操作 | 简单场景（有并发风险） |

### 75. 微服务如何做灰度发布？

- **Istio（Service Mesh）**：通过 VirtualService 定义流量路由规则，无侵入。

## 八、消息队列 Kafka（5 题）

### 76. Kafka 为什么吞吐量这么高？

**答案：**

1. **顺序写磁盘**：追加写入 log 文件，避免随机 I/O（顺序写速度接近内存）。
2. **零拷贝（Zero Copy）**：使用 `sendfile()` 系统调用，数据直接从 Page Cache 发送到网络，减少内核态到用户态的数据拷贝。
3. **Page Cache**：利用操作系统的页缓存，写入时不直接落盘，由 OS 管理刷盘。
4. **批量发送**：Producer 批量发送消息，减少网络请求。
5. **压缩**：支持 Snappy、LZ4、ZSTD 压缩，减少网络传输和磁盘占用。
6. **分区并行**：Topic 分为多个 Partition，不同 Partition 可并行读写。

---

### 77. Kafka 如何保证消息不丢失？

**答案：**

**Producer 端**：
- `acks=all`：所有 ISR 副本确认后才算发送成功。
- `retries > 0`：发送失败自动重试。
- `enable.idempotence=true`：开启幂等性，防止重试导致消息重复。

**Broker 端**：
- `replication.factor >= 3`：每个 Partition 至少 3 个副本。
- `min.insync.replicas >= 2`：ISR 中最少副本数为 2。
- `unclean.leader.election.enable=false`：禁止非 ISR 副本当选 Leader。

**Consumer 端**：
- 手动提交 offset（`enable.auto.commit=false`）。
- 消费完成业务逻辑后再提交 offset。
- 消费失败时不提交，进行重试或进入死信队列。

---

### 78. Kafka 如何保证消息的顺序消费？

**答案：**

**Kafka 的顺序保证**：
- **单 Partition 内有序**：Kafka 只保证同一个 Partition 内的消息有序。
- **全局有序**：Topic 只有一个 Partition（牺牲并发能力）。

**消费者端保证顺序**：
1. 相同业务 key 的消息发送到同一 Partition（使用相同的 partition key）。
2. 消费者单线程处理，或者按 partition key 分区到不同的处理线程。
3. 消费失败重试时，保证同一消息重试时不被其他消息插队（使用内存队列按 key 分区）。

**乱序处理场景**：
- 如果业务允许最终一致性，可以不保证严格顺序。
- 通过消息中的时间戳或版本号在消费端做排序。

---

### 79. Kafka 的消费者组（Consumer Group）是如何工作的？

**答案：**

**核心概念**：
- 同一个 Consumer Group 内的消费者共同消费一个 Topic，每个 Partition 只被组内一个消费者消费。
- 不同 Consumer Group 之间互不影响，各自独立消费。

**分区分配**：
- 消费者数量 <= Partition 数量：每个消费者消费一个或多个 Partition。
- 消费者数量 > Partition 数量：多余的消费者空闲。

**Rebalance（重平衡）**：
- **触发条件**：消费者加入/退出、Partition 数量变化、消费者被判定为"死亡"。
- **过程**：所有消费者放弃当前分区分配，重新协商分配方案。
- **影响**：Rebalance 期间消费暂停（Stop The World）。
- **优化**：
  - 合理设置 `session.timeout.ms`（消费者超时）。
  - 使用 `CooperativeStickyAssignor`（增量式 Rebalance，减少停顿）。
  - `Static Membership`（静态成员，短暂重连不触发 Rebalance）。

---

### 80. Kafka 的数据保留策略和日志清理机制是怎样的？

**答案：**

**数据保留策略**（按 Partition 独立配置）：
1. **基于时间**（默认）：`log.retention.hours=168`（7 天），超期的 log segment 被删除。
2. **基于大小**：`log.retention.bytes`，Partition 大小超过阈值时删除旧 segment。

**日志清理机制**：
- **Delete（默认）**：直接删除过期 segment。
- **Compact**：保留每个 key 的最新值，删除旧值（适用于 changelog 类数据，如 CDC）。

**日志分段（Log Segment）**：
- 每个 Partition 由多个 `.log` 文件组成（默认 1GB）。
- 配套的 `.index`（偏移量索引）和 `.timeindex`（时间索引）加速查找。
- 删除时以 segment 为单位。

## 九、分布式系统（10 题）

### 81. CAP 理论是什么？

**答案：**

**CAP 理论**：分布式系统最多只能同时满足以下三个特性中的两个：

| 特性 | 含义 |
|------|------|
| **C（Consistency）** 一致性 | 所有节点在同一时刻看到相同的数据 |
| **A（Availability）** 可用性 | 每个请求都能在合理时间内收到非错误响应 |
| **P（Partition Tolerance）** 分区容错性 | 网络分区发生时系统仍能运行 |

**实际选择**：
- **CP**：ZooKeeper、Nacos（CP 模式）、Etcd -- 网络分区时可能不可用，但数据一致。
- **AP**：Eureka、Nacos（AP 模式）、Cassandra -- 始终可用，但可能读到旧数据。
- **CA**：单节点数据库，不是分布式系统。

> 由于网络分区在分布式系统中不可避免，实际上是在 C 和 A 之间做权衡。

---

### 82. 分布式锁有哪些实现方式？

**答案：**

| 实现方式 | 原理 | 优点 | 缺点 |
|---------|------|------|------|
| **Redis** | SETNX + 过期时间 | 性能好、实现简单 | 主从切换可能丢锁 |
| **ZooKeeper** | 临时有序节点 | 强一致、自动释放 | 性能不如 Redis |
| **MySQL** | 唯一索引 / FOR UPDATE | 实现简单 | 性能差、无超时机制 |
| **Etcd** | Lease + 事务 | 强一致、高可用 | 运维成本 |

**Redis 分布式锁最佳实践**（Redisson）：
- 使用 Lua 脚本保证原子性。
- 看门狗自动续期（默认 30s，每 10s 续一次）。
- 联锁（MultiLock）：对多个 Redis 节点加锁。
- 读写锁：读锁可并发获取，写锁互斥。

### 83. 分布式 ID 生成方案有哪些？

**答案：**

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **UUID** | 128 位随机数 | 本地生成、无依赖 | 无序、性能差、存储大 |
| **数据库自增** | AUTO_INCREMENT | 简单、有序 | 单点瓶颈 |
| **号段模式** | 批量从数据库获取 ID 段 | 减少 DB 访问 | 依赖数据库 |
| **雪花算法（Snowflake）** | 时间戳 + 机器 ID + 序列号 | 高性能、趋势递增 | 时钟回拨问题 |
| **美团 Leaf** | 号段模式 + Snowflake | 双 Buffer 预分配 | 需部署服务 |
| **百度 UidGenerator** | Snowflake 变体 | 预生成、高性能 | 需数据库支持 |

**雪花算法结构（64 位）**：
```
0 - 41位时间戳(毫秒) - 10位机器ID - 12位序列号
```
- 41 位时间戳可用约 69 年。
- 10 位机器 ID 支持 1024 个节点。
- 12 位序列号每毫秒可产生 4096 个 ID。

### 84. 一致性 Hash 算法是什么？解决了什么问题？

**答案：**

**解决的问题**：传统取模 hash（`hash(key) % N`）在节点增减时，大量 key 需要重新映射（数据迁移量大）。

**原理**：
1. 将 hash 空间组织成一个环（0 ~ 2^32 - 1）。
2. 将节点和 key 都映射到环上（使用相同的 hash 函数）。
3. key 沿环顺时针方向找到第一个节点，由该节点负责。
4. 增加/删除节点时，只影响相邻节点间的 key。

**数据倾斜问题**：节点少时分布不均匀。解决：**虚拟节点**，每个物理节点对应多个虚拟节点，均匀分布在环上。

**应用场景**：
- Redis Cluster 的 hash slot（16384 个槽）。
- Dubbo 的一致性 Hash 负载均衡。
- 分布式缓存（Memcached 客户端）。

### 85. 分布式系统中的选举算法有哪些？

**答案：**

| 算法 | 应用场景 | 特点 |
|------|---------|------|
| **Raft** | Etcd、Consul、Nacos CP 模式 | 易于理解，Leader 选举 + 日志复制 |
| **ZAB** | ZooKeeper | 类似 Raft，支持崩溃恢复 |
| **Paxos** | Chubby | 理论完善但实现复杂 |
| **Gossip** | Redis Cluster、Cassandra | 去中心化，最终一致 |

**Raft 选举流程**：
1. 所有节点初始为 Follower。
2. Follower 超时未收到心跳，转为 Candidate，发起投票（term +1）。
3. 获得多数票的 Candidate 成为 Leader。
4. Leader 定期发送心跳维持权威。
5. 网络分区时少数派分区无法选出 Leader，只有多数派分区正常工作。

### 86. 什么是服务雪崩？如何防止？

**答案：**

**服务雪崩**：一个服务故障导致调用链上的多个服务级联失败，最终整个系统不可用。

**原因**：
- 服务间强依赖，没有容错机制。
- 线程池/连接池被占满。
- 重试风暴（大量重试加剧负载）。

**防止措施**：
1. **超时控制**：为每个远程调用设置合理超时。
2. **熔断器**：错误率达到阈值时快速失败，避免继续调用故障服务。
3. **降级**：返回缓存数据或默认值。
4. **限流**：控制入口流量和内部流量。
5. **隔离**：线程池隔离（每个服务使用独立线程池）、信号量隔离。
6. **异步解耦**：通过消息队列削峰填谷。
7. **多活/冗余**：服务多实例部署，避免单点故障。

### 88. BASE 理论是什么？和 ACID 有什么区别？

**答案：**

**BASE 理论**是对 CAP 中 AP 的延伸，是分布式系统的实践指导：

| 特性 | 含义 |
|------|------|
| **BA（Basically Available）** 基本可用 | 允许损失部分可用性（响应时间延长、降级） |
| **S（Soft State）** 软状态 | 允许数据存在中间状态（数据同步延迟） |
| **E（Eventually Consistent）** 最终一致性 | 经过一段时间后数据最终一致 |

**ACID vs BASE**：
| 特性 | ACID | BASE |
|------|------|------|
| **一致性** | 强一致 | 最终一致 |
| **可用性** | 低 | 高 |
| **适用场景** | 传统关系型数据库 | 分布式系统、NoSQL |
| **设计理念** | 追求正确性 | 追求可用性和性能 |

### 89. 什么是 Raft 算法？

**答案：**

Raft 是一个易于理解的分布式共识算法，等价于 Paxos 但更易实现。

**三个子问题**：
1. **Leader 选举**：
   - 任期（Term）：每个任期从选举开始。
   - Follower 超时未收到心跳 -> Candidate -> 发起投票。
   - 多数票胜出成为 Leader。
2. **日志复制**：
   - Leader 接收客户端请求，追加到本地日志。
   - 发送 AppendEntries RPC 给 Follower。
   - 多数 Follower 确认后，Leader 提交日志并通知 Follower 提交。
3. **安全性**：
   - 选举限制：只有拥有最新日志的 Candidate 才能当选。
   - Leader 完整性：一旦日志被提交，不会丢失。

---

### 90. 如何保证分布式系统中的数据一致性？

**答案：**

**方案选型**：
1. **强一致性**：
   - 2PC/3PC（分布式事务协议）。
   - Raft/Paxos（共识算法）。
   - 适用：金融交易、账户扣款。
2. **最终一致性**：
   - TCC（Try-Confirm-Cancel）：预留资源 -> 确认 -> 取消补偿。
   - SAGA：长事务拆分为多个本地事务 + 补偿事务。
   - 本地消息表 + 定时任务重试。
   - 事务消息（RocketMQ）：半消息 -> 本地事务 -> 提交/回滚消息。
   - 适用：大部分互联网业务。

**常用模式**：
- **可靠消息最终一致性**：发送方保证消息一定发出，消费方保证消息一定消费（幂等）。
- **最大努力通知**：通过多次重试通知（如支付结果回调），适合跨平台通知。

## 十、系统设计与场景题（5 题）

### 91. 如何设计一个秒杀系统？

**答案：**

**核心原则**：尽量将请求拦截在上游，减少到达数据库的请求量。

**架构设计**：
1. **前端**：按钮防重复点击、验证码、静态资源 CDN。
2. **网关层**：IP 限流、用户级限流、黑名单过滤。
3. **应用层**：
   - 请求排队（MQ 异步处理）。
   - Redis 预减库存（Lua 脚本原子操作）。
   - 库存不足直接返回，不再访问数据库。
4. **数据库**：
   - 乐观锁扣减：`UPDATE stock SET count = count - 1 WHERE id = ? AND count > 0`。
   - 分库分表降低单表压力。

**关键技术**：
- **Redis 预扣库存**：`DECR` 操作，库存不足时拒绝请求。
- **MQ 削峰**：秒杀请求入队，消费者按能力消费。
- **热点隔离**：秒杀服务独立部署，不影响其他业务。
- **兜底方案**：限流降级 + 服务熔断。

### 92. 如何设计一个短链系统？

**答案：**

**核心流程**：长链 -> 短码 -> 短链；访问短链 -> 查短码 -> 重定向到长链。

**短码生成方案**：
1. **自增 ID + Base62 编码**：ID = 12345 -> Base62 -> "3d7"（6 位 Base62 可表示 568 亿个）。
2. **MD5/MurmurHash + 取前 N 位**：冲突时追加或重试。
3. **发号器（Snowflake / 号段模式）**：分布式环境下生成唯一 ID。

**存储设计**：
```sql
CREATE TABLE short_url (
    id BIGINT PRIMARY KEY,
    short_code VARCHAR(10) UNIQUE,
    long_url TEXT,
    create_time DATETIME,
    expire_time DATETIME
);
```

**优化**：
- **缓存**：Redis 缓存热点短链映射。
- **302 重定向**：临时重定向（可统计访问次数），301 是永久重定向（浏览器缓存，无法统计）。
- **布隆过滤器**：拦截不存在的短码请求。

### 93. 线上 CPU 飙高如何排查？

**答案：**

**排查步骤**：
1. **找到高 CPU 的进程**：`top` 命令。
2. **找到高 CPU 的线程**：`top -Hp <pid>`。
3. **线程 ID 转为十六进制**：`printf '%x\n' <tid>`。
4. **导出线程堆栈**：`jstack <pid> > thread.log`。
5. **在堆栈中搜索线程 ID**：找到对应的代码位置。

**常见原因**：
- **死循环**：HashMap 并发导致链表成环（JDK 7）。
- **频繁 GC**：`jstat -gcutil <pid> 1000` 查看 GC 频率，Full GC 频繁说明内存不足或有内存泄漏。
- **正则回溯**：复杂正则表达式导致指数级回溯。
- **线程争用**：大量线程阻塞在锁上。

**工具**：Arthas（阿里开源，`thread -n 3` 查看 CPU 最高的 3 个线程）、JProfiler、async-profiler。

### 94. 如何处理接口响应慢的问题？

**答案：**

**排查流程**：
1. **确认慢在哪里**：
   - 链路追踪（SkyWalking/Zipkin）定位慢 Span。
   - APM 工具查看各层耗时（网关 -> 应用 -> 数据库 -> 缓存 -> 外部服务）。
2. **应用层**：
   - 线程池是否饱和。
   - GC 是否频繁（`jstat -gcutil`）。
   - 是否有锁竞争（`jstack` 查看 BLOCKED 线程）。
3. **数据库层**：
   - 慢查询日志。
   - 索引是否命中（EXPLAIN）。
   - 连接池是否耗尽。
4. **网络层**：
   - DNS 解析耗时。
   - 跨地域调用延迟。
   - 连接复用（Keep-Alive、连接池）。

**优化措施**：
- 引入缓存（Redis/本地缓存）。
- 异步化（MQ）。
- 并行调用（CompletableFuture）。
- 数据库优化（索引、读写分离、分页优化）。
- 连接池调优。
- 接口合并（BFF 层聚合多个服务数据）。

### 95. 线上频繁 Full GC 如何排查？

**答案：**

**排查步骤**：
1. **确认 GC 情况**：`jstat -gcutil <pid> 1000` 查看各代使用率和 GC 次数/时间。
2. **查看 GC 日志**：`-Xloggc:gc.log -XX:+PrintGCDetails`（JDK 8）或 `-Xlog:gc*:file=gc.log`（JDK 9+）。
3. **分析 Heap Dump**：
   - 配置 `-XX:+HeapDumpOnOutOfMemoryError`。
   - 使用 MAT 分析大对象和泄漏路径。
4. **常见原因**：
   - **内存泄漏**：集合持续增长不清理（如 static Map）、ThreadLocal 未清理。
   - **大对象**：一次性查询大量数据、导出 Excel。
   - **元空间不足**：动态代理生成大量类。
   - **新生代太小**：对象过早晋升到老年代。
5. **调优方向**：
   - 修复内存泄漏。
   - 调整堆大小和新生代比例。
   - 选择合适的 GC 回收器（G1、ZGC）。

## 十一、Spring AI & Java+AI 新方向（5 题）

### 96. Spring AI 是什么？它的核心架构是怎样的？

**答案：**

**Spring AI** 是 Spring 官方推出的 AI 应用开发框架（2025 年 5 月发布 1.0 GA），目标是为 Java 开发者提供统一的 AI 工程抽象。

**核心架构**：
1. **统一 API**：对多种 AI 提供商（OpenAI、Azure OpenAI、Ollama、阿里通义等）提供统一的接口抽象。
2. **ChatClient**：流式 API 封装大模型调用，支持 Prompt 模板、输出解析、工具调用。
   ```java
   ChatClient chatClient = ChatClient.builder(chatModel).build();
   String response = chatClient.prompt()
       .user("解释一下什么是微服务")
       .call()
       .content();
   ```
3. **Model 抽象**：`ChatModel`、`EmbeddingModel`、`ImageModel` 等。
4. **RAG 支持**：`DocumentReader` -> `DocumentTransformer`（切分） -> `DocumentWriter` -> `VectorStore`。
5. **Function Calling**：通过 `@Tool` 注解让大模型调用 Java 方法。
6. **MCP（Model Context Protocol）**：标准化的 AI 模型与外部工具/数据源交互协议。

### 97. Spring AI 和 LangChain4j 有什么区别？如何选型？

**答案：**

| 维度 | Spring AI | LangChain4j |
|------|-----------|-------------|
| **生态** | Spring 官方出品，与 Spring Boot 深度集成 | 社区驱动，LangChain 的 Java 移植 |
| **成熟度** | 1.0 GA（2025.5），较新 | 迭代到 1.x，更成熟 |

### 98. RAG（检索增强生成）的原理是什么？为什么需要切分文档？

**答案：**

**RAG 流程**：
```
文档 -> 切分（Chunking）-> 向量化（Embedding）-> 存入向量数据库
                                                    |
用户提问 -> 问题向量化 -> 向量检索相似文档 -> 拼接上下文 -> 大模型生成回答
```

**为什么需要切分文档**：
1. **Token 限制**：大模型有上下文窗口限制（如 4K/8K/128K Token），无法一次输入整个文档。
2. **检索精度**：更小的 chunk 语义更聚焦，检索更精准。
3. **成本控制**：减少无关内容输入，降低 Token 消耗和 API 费用。
4. **并行处理**：多个 chunk 可并行向量化。

**切分策略**：
- **固定大小切分**：按 Token 数或字符数切分（简单但可能破坏语义）。
- **语义切分**：按段落、章节等自然边界切分。
- **递归切分**：先尝试大粒度切分，不满足条件再细分。
- **重叠（Overlap）**：相邻 chunk 保留一定重叠，保持上下文连续性。

**Java 实现**：
- Spring AI：`TokenTextSplitter`、`JsonMetadataContentFilter`。
- LangChain4j：`DocumentSplitters.recursive()`。

### 99. MCP（Model Context Protocol）是什么？在 Java 中如何实现？

**答案：**

**MCP（Model Context Protocol）** 是由 Anthropic 提出的开放标准协议，定义了 AI 模型与外部工具/数据源之间的标准化交互方式。可以理解为 **AI 世界的 USB-C 接口**。

**核心概念**：
- **MCP Server**：提供工具（Tools）、资源（Resources）、提示模板（Prompts）的服务端。
- **MCP Client**：AI 应用作为客户端，发现并调用 MCP Server 提供的能力。
- **Transport**：通信方式，支持 Stdio（本地进程）和 HTTP+SSE（网络）。

**Java 中实现 MCP Server（Spring AI）**：
```java
@Service
public class WeatherService {
    @Tool(description = "查询指定城市的天气")
    public String getWeather(String city) {
        return city + ": 晴, 25C";
    }
}
```

**配置 Spring AI MCP Server**：
```yaml
spring:
  ai:
    mcp:
      server:
        name: weather-server
        version: 1.0.0
```

**应用场景**：
- 让大模型查询数据库、调用内部 API。
- 统一企业内部 AI 工具接入标准。
- IDE 插件（如 Cursor）通过 MCP 访问项目上下文。
